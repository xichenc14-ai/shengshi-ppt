## 一、设计背景与目标

### 1.1 用户需求

用户明确指出：**逻辑图形库是PPT领域的极其重要的核心能力**。用户需要：

1. **逻辑关系类型多样** — 平行关系、递进关系、时间序列、因果关系、对比关系、循环关系、层级关系等
2. **每种逻辑关系对应不同的逻辑图形/布局** — 不是简单列表，而是语义匹配的可视化方案
3. **充分调研参考** — 对标 Gamma、Beautiful.ai、Canva 等顶尖产品
4. **设计我们自己的逻辑图形库** — 不是抄，而是基于竞品分析设计适合国内用户习惯的系统

### 1.2 设计目标

| 目标 | 描述 |
|------|------|
| **覆盖度** | 支持 7+ 种核心逻辑关系类型 |
| **匹配精度** | 根据用户文本语义自动推荐最合适的图形 |
| **数量规模** | 每种逻辑关系至少 3+ 变体（不同节点数、不同布局方向） |
| **渲染质量** | SVG/Canvas 矢量渲染，支持缩放不变形 |
| **扩展性** | 支持新增逻辑关系类型和新的图形变体 |

## 二、竞品分析

### 2.1 Gamma

**优势：**
- 卡片式模板，每张卡片是一个完整的逻辑页面
- 逻辑关系通过 AI 自动识别并匹配合适的布局
- 支持嵌入图表、媒体、交互元素
- 模板库丰富，覆盖营销、汇报、教育等多个场景

**不足：**
- 国内访问不稳定
- 逻辑图形以整体页面为单位，不支持单独编辑逻辑组件
- 无明确的"逻辑图形库"概念，图形绑定在模板中

### 2.2 Beautiful.ai

**优势：**
- "Smart Slides"理念：每张幻灯片自动适配内容
- 逻辑图表分类清晰：时间轴、流程图、对比、层次结构
- 拖拽式编辑，图形组件化程度高

**不足：**
- 依赖云端，离线无法使用
- 逻辑图形数量有限，每种关系类型的变体较少
- 对中文排版支持一般

### 2.3 Canva

**优势：**
- 逻辑图表模板数量庞大（通过社区模板扩展）
- 视觉风格多样，从商务到创意均有覆盖
- 整合在设计生态中，可与其他设计资源配合

**不足：**
- 逻辑关系分类不系统，主要按视觉样式分类
- AI 辅助能力弱于 Gamma
- 演示文稿功能不是核心重点

### 2.4 国内传统PPT工具（WPS、Office）

**优势：**
- 基础逻辑图形齐全（流程图、时间轴等）
- 用户基数大，图形熟悉度高

**不足：**
- 图形样式老旧，缺乏现代设计感
- 不支持语义匹配，用户需手动选择
- 图形间风格不统一

## 三、逻辑关系类型体系

### 3.1 核心逻辑关系类型（8类）

```
逻辑关系类型
├── 并列关系（Parallel）
├── 递进关系（Progressive）
├── 时间序列（Timeline/Sequence）
├── 对比关系（Comparison）
├── 因果关系（Cause-Effect）
├── 循环关系（Cyclic）
├── 层级关系（Hierarchical）
└── 包含关系（Containment/Matrix）
```

### 3.2 每种关系的详细定义与图形映射

#### 3.2.1 并列关系（Parallel）

**定义：** 各要素之间是同一等级，无主次之分，顺序可调。

**典型场景：**
- 产品功能列表
- 团队成员介绍
- 关键指标展示
- 品牌/服务商展示

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 卡片列表 | 2-N | 水平/垂直 | 等宽卡片，横向或纵向排列 |
| 图标墙 | 4-12 | 网格/菱形 | 每项配图标，适合功能展示 |
| 双列对比 | 2 | 左右 | 左右各一项，适合两者并列 |
| 三列卡片 | 3 | 水平 | 三项横向排列，最常见 |
| 四宫格 | 4 | 2×2 | 四项均分，适合指标展示 |

#### 3.2.2 递进关系（Progressive）

**定义：** 各元素之间存在逻辑先后或程度深化关系，顺序固定，不可调换。

**典型场景：**
- 操作步骤（Step 1 → Step 2 → Step 3）
- 实施路径（规划 → 开发 → 落地）
- 成长阶段（启蒙 → 成长 → 成熟）
- 方法论（发现问题 → 分析问题 → 解决问题）

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 线性步骤条 | 2-5 | 水平 | 带编号的步骤，适合流程说明 |
| 垂直时间轴 | 3-8 | 垂直 | 左侧时间点，右侧内容 |
| 阶梯递进 | 3-6 | 斜向/垂直 | 阶梯式布局，视觉上体现"层层深入" |
| 漏斗图 | 3-5 | 垂直 | 从上到下逐步过滤，适合筛选类内容 |
| 箭头流程 | 3-6 | 水平/垂直 | 节点+箭头，体现方向性 |

#### 3.2.3 时间序列（Timeline）

**定义：** 按时间顺序排列的事件或发展阶段，有明确的时间节点。

**典型场景：**
- 公司发展历程
- 项目里程碑
- 历史事件回顾
- 成长经历

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 横向时间轴 | 3-10 | 水平 | 时间线在中间，上下分布内容 |
| 垂直时间轴 | 3-10 | 垂直 | 时间线在左侧，右侧展开内容 |
| 双轨时间轴 | 4-8 | 水平 | 上下两条时间线，适合对比时间线 |
| 里程碑点 | 3-8 | 水平 | 仅标注关键节点，省略中间细节 |
| 圆形时间轴 | 4-8 | 环形 | 环形排列，适合周期性回顾 |

#### 3.2.4 对比关系（Comparison）

**定义：** 两个或多个元素进行对照，突出差异或优劣。

**典型场景：**
- 方案A vs 方案B
- Before vs After
- 目标 vs 实际
- 竞品对比

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| VS对比 | 2 | 左右 | 经典"VS"样式，中间分割 |
| 四象限矩阵 | 4 | 2×2 | 四个区域，适合双维度对比 |
| 双柱对比图 | 2 | 垂直柱状 | 数据对比场景 |
| 表格式对比 | 2-4 | 表格 | 结构化对比，维度清晰 |
| 天平图 | 2 | 中心对称 | 视觉上体现"平衡/失衡" |

#### 3.2.5 因果关系（Cause-Effect）

**定义：** 一个或多个原因导致一个或多个结果，体现逻辑推导。

**典型场景：**
- 问题分析（原因 → 结果）
- 策略推导（输入 → 转化 → 输出）
- 逻辑链条（因为...所以...）

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 因果链条 | 2-5 | 水平 | 原因 → 结果，中间可有多级推导 |
| 鱼骨图（Ishikawa） | 1原因+6子项 | 鱼骨状 | 根本原因分析 |
| 漏斗因果 | 3-5 | 垂直 | 上宽下窄，体现"多因一果" |
| 树状推导 | 1→N | 向上/向下 | 从根因推导到结果 |
| 输入输出图 | 多入多出 | 中心 | 类似业务流程图，适合系统分析 |

#### 3.2.6 循环关系（Cyclic）

**定义：** 首尾相连，形成闭环，强调周期性或持续性。

**典型场景：**
- PDCA循环
- 产品迭代周期
- 生态循环
- 生命周期

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 环形循环 | 3-6 | 环形 | 标准循环图，节点沿圆周排列 |
| 三角形循环 | 3 | 等边三角 | 最简循环，三步迭代 |
| 四环循环 | 4 | 方形环 | 适合 PDCA 等四角循环 |
| 齿轮图 | 3-5 | 环形+齿轮 | 视觉强调"联动"感 |
| 太极循环 | 2 | 阴阳 | 适合二元对立又统一的关系 |

#### 3.2.7 层级关系（Hierarchical）

**定义：** 元素之间存在上下级或包含关系，体现结构层次。

**典型场景：**
- 组织架构
- 目标分解（OKR）
- 内容结构
- 权限层级

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 组织架构图 | 3-N | 树状 | 标准树形，金字塔或倒金字塔 |
| 单值金字塔 | 3-5 | 垂直 | 顶部最小，底部最大，体现包含关系 |
| 倒金字塔 | 3-5 | 垂直 | 顶部最大（总览），逐步细分 |
| 思维导图 | 1中心+N分支 | 放射状 | 适合发散性层级结构 |
| 嵌套层级 | 2-4 | 同心圆/嵌套 | 大环套小环，体现包含关系 |

#### 3.2.8 包含关系（Matrix）

**定义：** 元素之间存在包含或被包含关系，通常涉及多个维度。

**典型场景：**
- 业务分类（按维度切割）
- 组合分析（2×2、3×3矩阵）
- 要素拆解（总→分）
- SWOT分析

**推荐图形：**

| 图形名称 | 节点数支持 | 布局方向 | 说明 |
|---------|-----------|---------|------|
| 2×2矩阵 | 4 | 四象限 | 经典四宫格，如波士顿矩阵、SWOT |
| 3×3矩阵 | 9 | 九宫格 | 精细化分类 |
| 维恩图 | 2-4 | 圆环交集 | 体现交集关系 |
| 嵌套圆 | 2-4 | 同心圆 | 大圆包含小圆，适合"概念包含" |
| 表格矩阵 | M×N | 行列 | 结构化矩阵，适合多维度分析 |

## 五、语义匹配引擎

### 5.1 匹配流程

```
用户输入文本
    ↓
语义分析（关键词提取 + 关系识别）
    ↓
匹配逻辑关系类型（8类）
    ↓
根据节点数和上下文推荐最优布局
    ↓
用户确认/调整
    ↓
渲染输出
```

### 5.2 关键词权重表（部分）

| 逻辑类型 | 高权重关键词 | 中权重关键词 |
|---------|------------|------------|
| 并列关系 | 并列、同时、分别、以及、列表 | 同时进行、等价 |
| 递进关系 | 首先、然后、接着、最终、逐步 | 深入、递进、层层 |
| 时间序列 | 2020年、过去、历程、发展史 | 从...到...、时间线 |
| 对比关系 | VS、对比、差异、优劣、A与B | 相比、区别 |
| 因果关系 | 因此、导致、因为...所以...、结果 | 原因、使得 |
| 循环关系 | 循环、迭代、持续、周期、闭环 | 反复、周而复始 |
| 层级关系 | 包含、下属、分解、归类 | 从属、隶属 |
| 矩阵关系 | 四象限、分类、维度、矩阵 | 交叉、分组 |

### 5.3 匹配优先级规则

1. **关键词精确匹配** > 语义相似度匹配
2. **节点数约束**：如果用户输入内容可拆解为 N 个要点，优先推荐支持该节点数的布局
3. **上下文推断**：如检测到"项目"、"实施"等词，优先推荐流程类（递进/时间序列）
4. **用户历史偏好**：记录用户之前选择的布局类型，供后续推荐参考

## 六、技术实现方案

### 6.1 渲染技术选型

| 技术 | 适用场景 | 优缺点 |
|------|---------|--------|
| **SVG** | 静态图形、图标、简单布局 | ✅ 矢量无限缩放 ✅ 可交互 ✅ 体积小 ❌ 不适合复杂动画 |
| **Canvas** | 大量节点、高性能要求 | ✅ 性能好 ✅ 支持复杂图形 ❌ 位图放大失真 ❌ 交互复杂 |
| **HTML+CSS** | Web端展示、交互式编辑 | ✅ 开发快 ✅ 样式灵活 ❌ 导出为图片需额外处理 |
| **推荐：SVG为主，Canvas辅助** | 综合最优 | 矢量质量+可交互性 |

### 6.2 核心模块划分

```
LogicDiagramEngine
├── Parser           # 语义解析器
│   ├── keywordExtractor   # 关键词提取
│   ├── relationClassifier # 关系分类器
│   └── nodeCounter        # 节点数统计
│
├── Matcher          # 匹配引擎
│   ├── typeMatcher       # 类型匹配
│   ├── layoutSelector    # 布局选择
│   └── scorer            # 相似度打分
│
├── Renderer         # 渲染器
│   ├── SvgRenderer       # SVG输出
│   ├── CanvasRenderer    # Canvas输出（备选）
│   └── ComponentLib      # 图形组件库
│       ├── NodeShape     # 节点形状（rect/rounded/circle/diamond/capsule）
│       ├── Connector     # 连接线（straight/elbow/curve/arrow）
│       └── Label         # 标签样式
│
├── LayoutEngine     # 布局算法
│   ├── LinearLayout      # 线性布局
│   ├── RadialLayout      # 环形/放射布局
│   ├── TreeLayout        # 树状布局
│   └── GridLayout        # 网格布局
│
└── Exporter         # 导出器
    ├── PptxExporter      # 导出为PPTX（通过pptxgenjs）
    ├── SvgExporter       # 导出为SVG
    └── PngExporter       # 导出为PNG
```

### 6.3 API 接口设计

```typescript
// 语义分析 + 布局推荐
POST /api/diagram/parse
Request: {
  text: string              // 用户输入的文本内容
  options?: {
    maxNodes?: number       // 最大节点数限制
    preferredTypes?: string[] // 偏好类型
  }
}
Response: {
  detectedType: string      // 识别的逻辑关系类型
  nodeCount: number         // 检测到的节点数
  suggestedLayouts: [
    {
      layoutId: string
      name: string
      score: number         // 匹配度 0-1
      previewSvg: string   // 预览SVG
    }
  ]
}

// 渲染输出
POST /api/diagram/render
Request: {
  type: string              // 逻辑关系类型
  layoutId: string          // 选定的布局ID
  nodes: Array<{
    id: string
    label: string
    sublabel?: string
    icon?: string
  }>
  theme?: string            // 主题名称
  customColors?: string[]   // 自定义颜色
}
Response: {
  svg: string               // SVG字符串
  pptxUrl?: string          // PPTX下载链接（如需要）
}
```

### 6.4 技术实现路径

**Phase 1：基础框架（1-2周）**
- 设计逻辑关系类型数据模型（JSON Schema）
- 实现 8 种逻辑关系的 SVG 模板（每种至少 2 个布局）
- 基础语义解析器（关键词匹配）

**Phase 2：增强能力（2-3周）**

## 八、验收标准

### 8.1 功能验收

- [ ] 支持 8 种核心逻辑关系类型的识别和匹配
- [ ] 每种逻辑关系至少 3 个布局变体
- [ ] 语义匹配准确率 ≥ 85%（在标准测试集上）
- [ ] SVG 渲染输出清晰，支持 10x 放大不变形
- [ ] 导出为 PPTX 后图形保持可编辑性

### 8.2 性能验收

- [ ] 单次语义分析和布局推荐响应时间 < 500ms
- [ ] SVG 渲染时间 < 200ms（节点数 ≤ 10）
- [ ] 库文件总体积 < 5MB（不含图标资源）

### 8.3 体验验收

- [ ] 用户输入一段文本，系统自动推荐合适的图形
- [ ] 用户可查看不同布局变体的预览，选择最合适的
- [ ] 主题切换后图形样式整体协调

## 九、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 语义匹配准确率不足 | 用户体验差 | 先用关键词规则，积累数据后引入 ML 模型 |
| 布局变体数量不够 | 用户选择有限 | 优先完成每种类型 2 个核心布局，后续迭代扩展 |
| SVG 导出到 PPTX 格式丢失 | 图形变形 | 与 PPT 导出团队紧密配合，进行格式兼容性测试 |
| 中文排版效果差 | 用户抱怨 | 使用中文字体回退方案，确保字体可用 |

## 十、后续迭代方向

1. **AI 增强**：引入 LLM 对用户输入进行更精准的语义分析和布局推荐
