# 省心PPT 商业化架构评估报告

> **评估日期：** 2026-04-21  
> **版本：** v9.5.1  
> **域名：** shengxinppt.lol  
> **评估目标：** 10 → 100 → 1000 并发能力分析

---

## 📋 摘要（300字）

省心PPT当前架构（Next.js 16 + Vercel Serverless + Supabase + Gamma API）**可稳定支持10人并发**，100人并发需解决3个关键瓶颈，1000人并发需要架构级重构。

**核心发现：**
1. **Gamma API是最大瓶颈**：单key速率限制 burst=10000/hour=40000/day，但每次生成消耗~13-15 credits+轮询3分钟。10个key可支持100并发，但积分消耗是真正的成本瓶颈。
2. **LLM速度是用户体验瓶颈**：大纲生成用Kimi K2.5→MiniMax→GLM三级fallback，单次~15-53秒。用户感知"很慢"是正常的，需要改为streaming。
3. **Vercel Serverless不适合长任务**：outline API已设`maxDuration=60s`，Gamma生成靠前端轮询（设计合理），但Gamma直通模式的同步等待+Vercel冷启动会造成延迟。
4. **内存级rate-limit在多实例下失效**：Vercel每次冷启动创建新实例，内存中的`Map`无法跨实例共享，限流失效。

**成本估算：** 1000用户/天 × 1次生成 × ~13credits = 13,000 credits/天。Gamma Ultra plan ($39/月)约2000credits/月，**远不够**。需要Enterprise plan或大量credits购买。

## 1. 并发能力分析

### 1.1 10人同时使用 ✅ 足够

| 组件 | 状态 | 说明 |
|------|------|------|
| **Vercel前端** | ✅ 轻松应对 | CDN静态资源+边缘函数，10人零压力 |
| **API Routes** | ✅ 正常 | Vercel自动扩容，10个并发请求无压力 |
| **Gamma API** | ✅ 足够 | 单key: burst=10000/小时，10人×6次/小时=60次，远低于限制 |
| **LLM（Kimi/GLM）** | ✅ 正常 | 10人同时生成大纲，每个~15-45秒，队列自然消化 |
| **Supabase** | ✅ 正常 | Free tier: 500MB DB, 50,000 MAU, 10人无压力 |
| **限流** | ⚠️ 有缺陷 | 内存级限流在多实例下失效，但10人规模下问题不大 |

**结论：** 当前架构完全支持10人并发，无需改动。

### 1.2 100人同时使用 ⚠️ 需要改动

| 组件 | 状态 | 瓶颈分析 |
|------|------|---------|
| **Vercel前端** | ✅ 轻松 | CDN无压力 |
| **API Routes** | ⚠️ 注意冷启动 | Vercel Hobby有冷启动延迟(1-3s)，Pro plan可减少 |
| **Gamma API** | 🔴 **关键瓶颈** | 100人×6次/天=600次/天，单key足够但积分消耗大 |
| **LLM** | ⚠️ 排队延迟 | 100人同时请求大纲，Kimi+GLM可能排队，需15-60秒等待 |
| **Supabase** | ⚠️ 连接数 | Free tier: 60并发连接，Pro: 200。100人读写可能触及上限 |
| **Rate Limit** | 🔴 **失效** | 内存Map在多实例下完全失效，需要Redis或DB级限流 |

**需要的改动：**
1. **限流升级**：将内存限流迁移到Supabase/Redis（代码中已有L2 Supabase框架但未实现）
2. **Supabase升级**：从Free → Pro ($25/月)，200连接池
3. **Gamma Key池**：至少3-5个key轮换（代码已支持`GAMMA_API_KEYS`逗号分隔）
4. **LLM并发**：增加GLM API Keys数量（代码已支持`GLM_API_KEYS`轮换）

### 1.3 1000人同时使用 🔴 需要架构升级

| 组件 | 状态 | 瓶颈分析 |
|------|------|---------|
| **Vercel** | ⚠️ 成本 | Pro $20/月 + 用量，1000 MAU可能触发高用量费用 |
| **Gamma API** | 🔴 **成本爆炸** | 1000人×1次/天×13credits=13,000credits/天，~390,000/月 |
| **LLM** | 🔴 **排队严重** | 1000人高峰期，单模型排队时间>5分钟 |
| **Supabase** | 🔴 **连接瓶颈** | Pro plan 200连接不够，需要Team plan ($599/月) 或自建PG |
| **Rate Limit** | 🔴 **必须Redis** | 内存级完全失效，DB级查询延迟高 |

**需要的架构升级：**
1. **Gamma积分采购**：Enterprise plan或批量credits购买（详见第4节）
2. **独立后端**：从Vercel Serverless迁移到独立Node.js/容器服务（Railway/Fly.io/自建VPS）
3. **Redis**：Upstash（Vercel集成）或自建Redis做分布式限流+缓存
4. **数据库**：Supabase Team或自建PostgreSQL（连接池：PgBouncer）
5. **LLM队列**：引入消息队列（BullMQ + Redis），异步处理大纲生成
6. **CDN优化**：静态资源全量CDN，API走边缘函数

## 2. 关键技术瓶颈

### 2.1 Gamma API 速率限制

从文档和代码分析：

| Header | 值 | 说明 |
|--------|-----|------|
| `x-ratelimit-limit-burst` | 10,000 | 短时间窗口（约分钟级） |
| `x-ratelimit-limit` | 40,000 | 每小时 |
| `x-ratelimit-limit-daily` | 200,000 | 每天 |

**当前代码问题：**
- ✅ 已实现Key池轮换（`gamma-key-pool.ts`）
- ❌ **未读取rate-limit headers**进行自适应节流
- ❌ **未处理429错误**的指数退避
- ❌ 积分余额在内存中维护，**重启后丢失**（需要持久化到DB/Redis）

**并发能力推算（单key）：**
- 理论上限：200,000 requests/day
- 实际安全上限：~100,000 requests/day（留50%余量）
- 每次生成：1次POST + ~36次GET轮询(3分钟/5秒) = 37次请求
- 有效生成数：100,000 / 37 ≈ **2,700次生成/天/key**

**5个key：~13,500次生成/天**，足够1000用户×1次/天。

### 2.2 积分成本

|---------|------------|------|
| 基础PPT(8-10页,无AI图) | ~13-15 | text generation + pictographic |

| 大型PPT(20+页,AI图) | ~60-100 | 页数×2 + AI图 |

**用户侧积分定价（membership.ts）：**
- 免费：50 credits/注册
- 省心会员：300 credits = ¥9.9/月
- 高级会员：1000 credits = ¥29.9/月
- 尊享会员：2000 credits = ¥49.9/月

**平台侧成本（Gamma API credits ≠ 用户积分）：**
- 用户1积分 ≠ Gamma 1 credit（代码中`convertCreditsToUserPoints`是1:1）
- 实际上Gamma 1次生成≈13 Gamma credits，用户扣20积分(10页×2)
- **关键问题：Gamma credits和用户积分的换算关系需要明确**

**成本估算（1000用户/天）：**

### 2.3 Vercel Serverless 函数超时

| Plan | 函数超时 | 当前使用 |
|------|---------|---------|
| Hobby | 10秒 | ❌ outline API需60秒 |
| Pro | 60秒 | ✅ outline已设`maxDuration=60` |
| Enterprise | 300秒 | ✅ |
| 自建 | 无限制 | ✅ |

**当前设计分析：**
- ✅ **Gamma生成已正确使用异步模式**：POST创建→前端轮询GET，不占Serverless时间
- ⚠️ **outline API**：设了`maxDuration=60`，Kimi(45s)+MiniMax(30s)+GLM(30s)三级fallback，可能超时
- ⚠️ **smart-outline API**：未设`maxDuration`，默认10秒(Hobby)或60秒(Pro)，**可能超时**

**建议：** 所有涉及LLM调用的API route必须显式设置`export const maxDuration = 60`。

### 2.4 LLM 处理速度

| 模型 | 任务 | 预计耗时 | 代码中的超时 |
|------|------|---------|------------|
| Kimi K2.5 | 大纲生成 | ~15-20s | 45s |
| MiniMax M2.7 | Fallback | ~20-30s | 30s |
| GLM-5 | Last resort | ~15s(代码注释) | 无显式超时 |
| GLM-5-turbo | 直接生成 | ~19s(注释) | 无显式超时 |
| GLM-5.1 | 复杂任务 | ~53s(注释) | 无显式超时 |

**用户感知"很慢"的原因：**
1. 大纲生成：同步等待 → 用户看到loading → 15-60秒无反馈
2. Gamma生成：前端轮询(3秒间隔) → 但Gamma侧生成需要1-3分钟
3. **无streaming**：大纲生成是同步返回，用户看不到逐字输出

**建议：**
- Phase1：添加进度条（已有`GenerationProgress`组件，但需要更细粒度）
- Phase2：大纲生成改为streaming（SSE）
- Phase3：考虑本地缓存常见主题大纲

### 2.5 Supabase/PostgreSQL 连接数

| Plan | 最大连接数 | 并发限制 |
|------|----------|---------|
| Free | ~60 | 10人并发足够 |
| Pro | 200 | 100人并发可能紧张 |
| Team | 200+ PgBouncer | 1000人可行 |

**当前代码的连接模式：**
- 每个API route独立创建`createClient`（非单例）
- Supabase JS v2内部有连接池，但多个Serverless实例 = 多个连接池
- **Vercel Serverless每次冷启动创建新实例 → 新连接池**

**估算：**
- 10个Serverless实例 × 默认池大小(~10) = 100个连接
- Pro plan(200连接)可支持~20个实例
- 100人并发可能触发10+实例 → 需要Pro plan + 连接池优化

### 2.6 Redis 连接和内存

**当前状态：** 代码中**没有实际使用Redis**。
- `rate-limit.ts`用内存Map（单实例有效）
- Session用Iron Session（cookie加密，无服务端存储）
- 没有缓存层

**需要Redis的场景：**
- 分布式限流（100+并发必须）
- Gamma Key池状态持久化
- 大纲生成结果缓存
- 用户Session共享（如果多实例部署）

## 4. 积分配套建议

### 4.1 用户积分 vs Gamma Credits 的关系

**当前代码中的定义：**
- 用户积分：1积分 = 1 Gamma credit（`convertCreditsToUserPoints`）
- 用户扣费：10页PPT = 20用户积分（10页 × 2积分/页）
- Gamma实际消耗：~13-15 credits/次（基础PPT）

**问题：** 用户20积分 vs Gamma 13-15 credits，有~5积分的缓冲。但AI图PPT的Gamma消耗远高于用户扣费：
- AI图(6张)：Gamma消耗 ~25-30 credits，用户扣费 20 + 12(AI图) = 32积分
- 高级AI图(6张)：Gamma消耗 ~55-75 credits，用户扣费 20 + 60(AI图) = 80积分

**结论：** 当前定价基本覆盖Gamma成本，但需要根据实际消耗微调。

### 4.2 并发级别下的积分消耗

| 并发级别 | 假设 | Gamma Credits/月 | 需要的Gamma Plan |
|---------|------|-----------------|-----------------|
| **10人** | 10人×2次/周×4周 | ~1,040 | 1个Ultra($39/月) ✅ |
| **100人** | 100人×2次/周×4周 | ~10,400 | 5个Ultra($195/月) ⚠️ |
| **1000人** | 1000人×1次/周×4周 | ~52,000 | Enterprise(需谈判) 🔴 |
| **1000人(高频)** | 1000人×3次/周×4周 | ~156,000 | Enterprise + 批量购买 🔴 |

> 注意：以上是**基础PPT(无AI图)**的估算。含AI图的消耗翻倍。
